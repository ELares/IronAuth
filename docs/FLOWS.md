# The headless flow contract (issue #84)

IronAuth serves ONE machine readable flow object, consumed identically by hosted
pages, custom SPAs, and native apps, with only the transport varying. This is the
flagship developer experience surface: every M9 UI surface and every M12 SDK binds
to it. This document names the published, versioned artifacts and the rules that
keep them stable.

## The normative artifact

`docs/flow-schema.json` is the **normative** flow object contract: the JSON Schema
for the `Flow` object, generated from the Rust type by
`crates/ironauth-oidc/examples/flow-schema.rs`. It is the single source of truth an
SDK or a custom UI validates against. It is regenerated by `scripts/flow-schema.sh`
and its freshness is gated in CI, so the committed file can never drift from the
code.

Two companion artifacts accompany it, both generated and CI gated the same way:

- `docs/flow-messages.json` is the numeric message id registry: every stable
  numeric message id with its symbolic name, kind, default `en` text, and context
  keys. Internationalization keys on the numeric id and the structured context,
  never on the copy string. A changed or removed id fails the freshness gate; new
  ids are additive.
- `docs/flow-golden.json` is the golden flow object corpus: a byte stable set of
  representative flow objects, one per journey per state, on both transports. It
  locks the RENDERED shape the engine emits (the exact node set, the deterministic
  node order, the node groups, and the attached message ids), which the schema
  alone does not pin. It is regenerated by `scripts/flow-golden.sh` and gated in CI.

## Contract versioning

The flow object carries an integer `contract_version` (currently `1`), mirrored on
the wire as the `X-IronAuth-Flow-Contract` response header. The version is a single
monotonic integer, bumped ONLY on a schema breaking change.

The rule:

- An **additive** change (a new node group, a new message id, a new journey, a new
  optional field) does NOT bump `contract_version`. It updates the golden corpus and
  the schema and message registries. Existing clients keep working, so the golden
  and schema diffs are additive and no version change is warranted.
- A **breaking** change (a removed or renamed field, a changed field type, a removed
  node group or message id, a changed node ordering) MUST bump `contract_version` in
  `crates/ironauth-oidc/src/flow/model.rs`, update the mirrored header, and be
  recorded in the changelog below.

The gate that makes an accidental break LOUD:

1. The **schema freshness gate** (`scripts/flow-schema.sh` plus `git diff
   --exit-code`) fails any PR whose flow object type changed without regenerating
   `docs/flow-schema.json`. Regenerating produces a visible schema diff a reviewer
   reads.
2. The **golden corpus gate** (`scripts/flow-golden.sh` plus `git diff --exit-code`,
   and the `flow_matrix` corpus test) fails any PR that changes a journey's rendered
   flow without updating `docs/flow-golden.json`. A field removal, a reordering, or a
   changed message id all trip this diff, so the change cannot land silently.
3. The **contract version consistency test** (`flow_matrix`) fails unless the
   `contract_version` in the golden corpus, the message registry, and the
   `CONTRACT_VERSION` constant all agree, so a bump must be applied coherently across
   every artifact.

Distinguishing additive from breaking, and therefore whether the version bumps, is a
reviewer decision guided by this rule. The three gates guarantee that EVERY contract
change is loud and deliberate; the version bump is the author's explicit
acknowledgment that the change is breaking.

## One object, two transports

There is no Kratos style browser versus API bifurcation. The state machine, the node
assembly, the message ids, the error shaping, and the anti enumeration recipe are one
code path. The two transports differ only at two mechanical edges: submission
ingestion (a form post with the same origin CSRF check versus an `application/json`
body with a per flow submit token) and continuation (a cookie plus redirect versus a
JSON envelope). For any journey and state, the API and browser render the SAME flow
object once three documented deltas are normalized out: the `transport` tag, the
`ui.action`, and the browser only hidden `flow` node. The `flow_matrix` test proves
this equivalence for every state on both transports, and that ONE generic driver
walks every journey by following the flow object's nodes, with no journey specific
client logic.

## Contract changelog

- **v1** (issue #84): the initial published contract. Login, registration, MFA
  (challenge and enrollment), recovery, and the federation launcher journeys; typed
  UI nodes in deterministic order; numeric message ids with structured context;
  node level and flow level errors; transient payload passthrough; typed flow errors
  (expired, already completed, invalid submission, malformed transient payload).
