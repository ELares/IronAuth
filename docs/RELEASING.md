# Releasing

IronAuth is a monorepo with independently versioned artifacts. Every release
names exactly one artifact; there is no repo-wide release train, so an urgent
server security release never drags SDK churn along, and vice versa.

## One version scheme

An artifact release is a git tag `<artifact>-vX.Y.Z` where `<artifact>` is the
artifact's name in docs/COMPATIBILITY.md (generated by
scripts/compat-matrix.sh; CI fails if it is stale) and `X.Y.Z` equals the
version in that artifact's manifest at the tagged commit. The release workflow
verifies the equality and refuses to run on mismatch. There are no other
version identities: no marketing versions, no module-path versions that
diverge from tags. A version referenced anywhere (advisory, changelog, docs)
is always the manifest version of one named artifact.

Current artifact lanes:

- `ironauth-vX.Y.Z`: the server binary. Produces a static musl binary, a
  CycloneDX SBOM, GitHub build provenance attestations, a GitHub release with
  the crate's changelog section, and a cosign-signed container image on GHCR
  (`.github/workflows/release.yml`).
- Library crates (`ironauth-env`, and later crates) release to crates.io under
  their own `<crate>-vX.Y.Z` tags once their APIs stabilize; until then they
  version inside the workspace only.
- TypeScript artifacts (SDKs, admin SPA, hosted pages) get their own lanes
  when the first package lands (see packages/README.md).

## Changelogs

Every artifact keeps its own `CHANGELOG.md` next to its manifest with a
section per released version, newest first, and an Unreleased section that
release PRs must update. The release workflow extracts the tagged version's
section as the release notes.

## Security advisory format

Every security advisory (GitHub Security Advisory and the release notes entry)
must name, without exception:

1. **Artifact**: the exact artifact name from docs/COMPATIBILITY.md.
2. **Affected versions**: a closed version range per artifact, for example
   `ironauth >= 0.3.0, < 0.7.2`.
3. **Patched version**: the first version containing the fix, per artifact.
4. **Severity**: CVSS score plus a one-line plain-language impact statement.

Advisories that touch multiple artifacts list every (artifact, range, patched)
triple separately. "Update to the latest version" without a named version
range is not an acceptable advisory body. Security-relevant fixes are also
called out with severity in the release notes of the patched release, never
silently included.

## Release checklist

1. PR updating the artifact's manifest version and its `CHANGELOG.md`
   (moving Unreleased content under the new version heading).
2. Merge on green; tag `<artifact>-vX.Y.Z` on the merge commit; push the tag.
3. The release workflow runs verify, build, SBOM, attest, release, and the
   image lane; check all lanes finished and the release page lists the
   binary, SHA256SUMS, and SBOM.
4. First-time note: the release lane is validated end to end by the first
   real release; until then treat it as authored but unproven.
